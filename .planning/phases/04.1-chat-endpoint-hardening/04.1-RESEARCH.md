## User Constraints (from CONTEXT.md)

### Locked Decisions

#### Auth enforcement

- Chat endpoint requires authentication regardless of global AUTH_MODE setting
- Handler-level check: chat handler verifies user identity is non-nil before proceeding
- Returns JSON 401 with message "authentication required for AI advisor" for unauthenticated requests
- No role restriction -- any authenticated user can chat (no 'operator' role gate)
- Consistent with existing pre-stream JSON error pattern (same as nil provider, bad request, etc.)

#### Scenario state lifecycle

- Per-user scenario storage keyed on authenticated username from auth context
- Handler gets a map[string]\*models.ScenarioComparison field (mutex-protected)
- CompareScenario stores the result after computing it, keyed to the requesting user
- buildChatSystemPrompt reads the authenticated user's last scenario (if any)
- Scenarios persist in memory until server restart -- no TTL
- Scenario endpoint keeps its current auth behavior (no new auth enforcement there)
- If a user isn't authenticated when running a scenario, the result is not stored (chat requires auth anyway, so it would never be read)

#### Provider defaults

- Add AI_MODEL env var for configurable model selection, default to Claude Sonnet (claude-sonnet-4-5-20250514)
- Accept any string for model name -- pass through to SDK, let Anthropic API validate
- MaxTokens hardcoded default at 4096 (not configurable via env var)
- main.go passes explicit ChatConfig with Model and MaxTokens to NewAnthropicProvider
- Config struct gets new AIModel field loaded from AI_MODEL env var

## Summary

This phase closes three tech debt items found in the v1.0 milestone audit. All three are straightforward code changes to existing files with no new dependencies, no new packages, and no architectural changes.

**Gap 1 (MISS-01 / CTX-03):** The `CompareScenario` handler computes a `ScenarioComparison` and returns it to the client but never stores it. The `buildChatSystemPrompt` method constructs a `ContextInput` that always leaves `Scenario` as nil. Fix: add a per-user `map[string]*models.ScenarioComparison` field (mutex-protected) to Handler, write to it in CompareScenario, read from it in buildChatSystemPrompt using the authenticated username.

**Gap 2 (GAP-01 / CHAT-02):** The chat route is protected by the global auth middleware, but when `AUTH_MODE=optional` the middleware passes anonymous requests through. The chat handler needs a handler-level guard: check `middleware.GetUserClaims(r) == nil` and return 401 JSON before any other processing.

**Gap 3 (GAP-02 / PROV-02):** `main.go` passes `ai.ChatConfig{}` (zero-value) to `NewAnthropicProvider`. Fix: add `AIModel` field to config, load from `AI_MODEL` env var (default `claude-sonnet-4-5-20250514`), pass `ai.ChatConfig{MaxTokens: 4096, Model: cfg.AIModel}`.

**Primary recommendation:** Three targeted changes to existing files. No new files needed. Each gap maps cleanly to one logical changeset.

## Architecture Patterns

### Pattern 1: Handler-Level Auth Guard (GAP-01)

**What:** Check `middleware.GetUserClaims(r)` at the top of the handler, before any business logic.
**When to use:** When a specific endpoint needs stricter auth than the global middleware provides.
**Current pattern in codebase:** The chat handler already has pre-stream validation checks (nil provider, empty messages, invalid roles) that return JSON errors via `h.writeError()`. The auth check slots in as the first check, before the nil provider check.

```go
// In chat.go, at the very top of Chat():
func (h *Handler) Chat(w http.ResponseWriter, r *http.Request) {
    // Auth guard: AI advisor requires authentication regardless of global AUTH_MODE
    claims := middleware.GetUserClaims(r)
    if claims == nil {
        h.writeError(w, "authentication required for AI advisor", http.StatusUnauthorized)
        return
    }

    // Existing nil provider check follows...
    if h.chatProvider == nil {
        h.writeError(w, "AI advisor not configured", http.StatusServiceUnavailable)
        return
    }
    // ...rest of handler
}
```

### Pattern 2: Per-User State on Handler Struct (MISS-01)

**What:** Store per-user results on the Handler struct behind a mutex, keyed by username.
**When to use:** When shared in-memory state needs to be accessible across handlers without passing through HTTP.
**Existing pattern:** Handler already has `infrastructureState *models.InfrastructureState` with `infraMutex sync.RWMutex`. The scenario map follows the same structure but is keyed per-user.

```go
// In handlers.go, add to Handler struct:
type Handler struct {
    // ...existing fields...
    userScenarios      map[string]*models.ScenarioComparison
    userScenariosMutex sync.RWMutex
}

// In NewHandler, initialize the map:
h := &Handler{
    // ...existing fields...
    userScenarios: make(map[string]*models.ScenarioComparison),
}
```

### Pattern 3: Write in CompareScenario, Read in buildChatSystemPrompt

**What:** CompareScenario stores the result; buildChatSystemPrompt reads it.
**When to use:** Cross-endpoint state sharing.

```go
// In scenario.go, after computing comparison:
comparison := h.scenarioCalc.Compare(*state, input)
comparison.Recommendations = models.GenerateRecommendations(*state)

// Store for authenticated users
claims := middleware.GetUserClaims(r)
if claims != nil {
    h.userScenariosMutex.Lock()
    h.userScenarios[claims.Username] = &comparison
    h.userScenariosMutex.Unlock()
}

h.writeJSON(w, http.StatusOK, comparison)
```

```go
// In chat.go, buildChatSystemPrompt now accepts username:
func (h *Handler) buildChatSystemPrompt(username string) string {
    input := ai.ContextInput{
        BOSHConfigured:    h.boshClient != nil,
        VSphereConfigured: h.cfg.VSphereConfigured(),
    }
    // ...existing dashboard/infra reads...

    // Read per-user scenario
    h.userScenariosMutex.RLock()
    input.Scenario = h.userScenarios[username]
    h.userScenariosMutex.RUnlock()

    ctx := ai.BuildContext(input)
    return ai.BuildSystemPrompt(ctx)
}
```

### Pattern 4: Explicit Provider Defaults (GAP-02)

**What:** Pass explicit model and max_tokens when constructing the Anthropic provider.
**Existing pattern:** Config struct loads env vars with defaults in `config.Load()`.

```go
// In config.go, add to Config struct:
AIModel string

// In Load():
AIModel: getEnv("AI_MODEL", "claude-sonnet-4-5-20250514"),

// In main.go:
chatProvider := ai.NewAnthropicProvider(cfg.AIAPIKey, ai.ChatConfig{
    MaxTokens: 4096,
    Model:     cfg.AIModel,
})
```

### Anti-Patterns to Avoid

- **Storing scenario in cache:** Don't use the cache package for per-user scenario state. The cache has TTL and is keyed by string, which would require constructing cache keys and dealing with expiry. A simple map is cleaner for ephemeral, non-expiring, per-user state.
- **Adding auth middleware to route definition:** The decision is handler-level enforcement, not changing the route's `Public` flag. The route already passes through the global auth middleware; the gap is that `optional` mode allows anonymous through.
- **Making MaxTokens configurable via env:** The decision explicitly says MaxTokens is hardcoded at 4096, not configurable. Don't add an env var for it.

## Common Pitfalls

### Pitfall 1: Auth Check Ordering in Chat Handler

**What goes wrong:** Auth check placed after the body read or SSE setup, causing partial processing before the 401 is returned.
**Why it happens:** Developers add new checks at the end of existing validation chains.
**How to avoid:** Auth check must be the very first check in Chat(), before even the nil provider check. The JSON error response pattern is identical to existing pre-stream checks.
**Warning signs:** Test passes but unauthenticated requests get 503 (nil provider) instead of 401.

### Pitfall 2: Deadlock Between scenarioMutex and infraMutex

**What goes wrong:** Acquiring locks in different orders across handlers causes deadlock.
**Why it happens:** CompareScenario reads infraMutex (existing) then writes userScenariosMutex (new). buildChatSystemPrompt reads both.
**How to avoid:** Consistent lock ordering -- always acquire infraMutex before userScenariosMutex. In buildChatSystemPrompt, read infra state first (release lock), then read scenario (separate lock acquisition). Never hold both locks simultaneously.
**Warning signs:** Server hangs under concurrent scenario-compare + chat requests.

### Pitfall 3: Zero-Value MaxTokens Still Reaching SDK

**What goes wrong:** Even with defaults set in NewAnthropicProvider, a per-request `WithMaxTokens(0)` could override to zero.
**Why it happens:** resolveConfig uses zero-value check: `if cfg.MaxTokens == 0 { cfg.MaxTokens = p.defaults.MaxTokens }`.
**How to avoid:** Not a problem for this phase -- no per-request MaxTokens option is used. The resolveConfig fallback works correctly because the chat handler only uses `WithSystem()`. Document this as a property: provider defaults are the effective defaults for MaxTokens and Model.

### Pitfall 4: Forgetting to Initialize userScenarios Map

**What goes wrong:** Nil map panic when CompareScenario attempts to write.
**Why it happens:** Map not initialized in NewHandler.
**How to avoid:** Initialize in NewHandler: `userScenarios: make(map[string]*models.ScenarioComparison)`.
**Warning signs:** Nil pointer dereference at first scenario comparison.

### Pitfall 5: buildChatSystemPrompt Signature Change Breaking Tests

**What goes wrong:** Adding `username` parameter to `buildChatSystemPrompt` breaks existing test call sites.
**Why it happens:** Tests call buildChatSystemPrompt indirectly through Chat handler, which works. But `TestChat_SystemPromptIncludesContext` sets up the handler without auth context.
**How to avoid:** In Chat handler, extract username from claims (already validated non-nil by auth guard) and pass to buildChatSystemPrompt. Tests that exercise Chat need to include user claims in the request context using `middleware.WithUserClaims()`.
