<decisions>
## Implementation Decisions

### Auth enforcement

- Chat endpoint requires authentication regardless of global AUTH_MODE setting
- Handler-level check: chat handler verifies user identity is non-nil before proceeding
- Returns JSON 401 with message "authentication required for AI advisor" for unauthenticated requests
- No role restriction -- any authenticated user can chat (no 'operator' role gate)
- Consistent with existing pre-stream JSON error pattern (same as nil provider, bad request, etc.)

### Scenario state lifecycle

- Per-user scenario storage keyed on authenticated username from auth context
- Handler gets a map[string]\*models.ScenarioComparison field (mutex-protected)
- CompareScenario stores the result after computing it, keyed to the requesting user
- buildChatSystemPrompt reads the authenticated user's last scenario (if any)
- Scenarios persist in memory until server restart -- no TTL
- Scenario endpoint keeps its current auth behavior (no new auth enforcement there)
- If a user isn't authenticated when running a scenario, the result is not stored (chat requires auth anyway, so it would never be read)

### Provider defaults

- Add AI_MODEL env var for configurable model selection, default to Claude Sonnet (claude-sonnet-4-5-20250514)
- Accept any string for model name -- pass through to SDK, let Anthropic API validate
- MaxTokens hardcoded default at 4096 (not configurable via env var)
- main.go passes explicit ChatConfig with Model and MaxTokens to NewAnthropicProvider
- Config struct gets new AIModel field loaded from AI_MODEL env var

### Claude's Discretion

- Exact mutex strategy for per-user scenario map (sync.RWMutex or sync.Map)
- Whether to add AI_MODEL to the health endpoint response
- Test fixture model strings

</decisions>
