RED phase: Add targeted test functions to `context_test.go`:

**TestBuildContext_CredentialSafety:**
Populate a `ContextInput` with realistic data where hostnames and deployment names appear (these are allowed). Then define a list of sentinel credential values matching every credential field in `config.Config`:

- `"CREDENTIAL_CF_PASSWORD_VALUE"` (CFPassword)
- `"CREDENTIAL_BOSH_SECRET_VALUE"` (BOSHSecret)
- `"CREDENTIAL_BOSH_CA_CERT_VALUE"` (BOSHCACert)
- `"CREDENTIAL_CREDHUB_SECRET_VALUE"` (CredHubSecret)
- `"CREDENTIAL_VSPHERE_PASSWORD_VALUE"` (VSpherePassword)
- `"CREDENTIAL_OAUTH_CLIENT_SECRET_VALUE"` (OAuthClientSecret)
- `"CREDENTIAL_AI_API_KEY_VALUE"` (AIAPIKey)

The test does NOT pass config.Config to BuildContext (that's the point -- the function signature prevents it). Instead, the test verifies:

1. None of the sentinel strings appear in the output (belt-and-suspenders: even if someone accidentally embedded one in a model field, it would be caught)
2. Hostnames and deployment names DO appear in the output (topology is allowed)
3. The function signature accepts only `ContextInput` (compile-time guarantee -- this test documents the intent)

Embed sentinel values in model fields where someone might accidentally copy a credential (e.g., set a cell Name to include "CREDENTIAL_CF_PASSWORD_VALUE" to prove the test catches it if it leaks). Then assert the output does NOT contain any sentinel values. This is intentionally adversarial -- the test proves the sentinel detection works.

Note: Since BuildContext doesn't accept config.Config, the primary credential isolation is structural (enforced by the type system). This test is the secondary safety net per CONTEXT.md: "Unit test uses distinctive sentinel values to verify no config credential fields appear in output."

**TestBuildContext_SegmentAggregation:**
Create a dashboard with cells across 3 segments: "shared" (empty string), "iso-seg-1", and "iso-seg-2". Include cells with known memory values. Assert that:

- Per-segment cell counts in output match expected counts
- Per-segment total memory in output matches sum of cell MemoryMB values
- Overall totals match the sum across all segments
- Empty-string segment appears as "shared" in output

**TestBuildContext_MarkerCompleteness:**
For each possible data-source state (configured+available, configured+unavailable, not configured), verify the output contains exactly the expected marker string. Test all combinations:

- BOSH not configured: "BOSH: NOT CONFIGURED"
- BOSH configured, unavailable: "BOSH: UNAVAILABLE"
- BOSH configured, available: "BOSH: available"
- vSphere not configured: "vSphere: NOT CONFIGURED" (or "NOT CONFIGURED" in the Infrastructure section)
- vSphere configured, unavailable: Infrastructure section has "UNAVAILABLE"
- vSphere configured, available: Infrastructure section has cluster data
  Assert that the Infrastructure section heading ALWAYS appears (even when vSphere is not configured).
  Assert that the Scenario Comparison section heading ALWAYS appears (even when no scenario has been run).

**TestBuildContext_TokenBudget:**
Create realistic-sized input: 50 apps, 3 isolation segments with 5-8 cells each, 2 clusters, a scenario comparison. Call BuildContext and measure the output length. Assert it's under 5000 characters (rough proxy for ~1000 tokens at 5 chars/token). This is a spot-check, not a precise measurement.

Run tests: `go test ./backend/services/ai/ -run "TestBuildContext_(CredentialSafety|SegmentAggregation|MarkerCompleteness|TokenBudget)" -v`. Tests should FAIL or PASS depending on Plan 02-01's implementation. If they pass immediately (because 02-01 already handles these cases), that's fine -- skip the GREEN phase for those tests. If any fail, proceed to Task 2.
